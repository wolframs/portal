# PORTAL RFC-003 — Inline attachment bytes (uploads from remote clients)

- **Status:** ✅ Implemented (2026-06-14)
- **Author:** Antra (drafted with Claude Code)
- **Date:** 2026-06-14
- **Affects:** `portal-protocol` (`rpc.ts` — `OutgoingFile`), `portal-relay` (`discord-bot.ts` `buildAttachments`, `config.ts`), `portal-client` (`fileFromBytes` helper), `portal-mcpl` (tool schema)

> **Implementation notes (deviations from the draft):**
> - **Size limit is per-*message* total decoded bytes** (`maxInlineFileBytes`,
>   default 8 MiB), not per-file. It matches Discord's own per-message upload cap
>   and *automatically* keeps the WS frame + relay memory bounded — so no
>   `gateway.ts` `maxPayload` change was needed (10×per-file could exceed the ws
>   100 MiB default; a per-message budget can't).
> - **`allowPathFiles` defaults false** (`PORTAL_ALLOW_PATH_FILES=true` to enable)
>   — closes the arbitrary-file-read hole on shared/multi-tenant relays.
> - Validation throws **`INVALID_PARAMS`** (not `INTERNAL`).
> - `portal-client` exports `fileFromBytes(name, data, opts?)`; the `portal-mcpl`
>   `send_message` tool schema now documents `bytes`/`name`.
> - **Verified:** unit (`test/attachments.test.ts` — bytes→attachment, missing
>   name, both/neither, path-disabled, per-message budget, >10 files) and live
>   (`scripts/p1-live.mjs`): a 1×1 PNG sent via `bytes` round-trips through
>   `fetch_history` as a `dot.png` attachment, and `{path:'/etc/passwd'}` is
>   rejected.
- **Protocol version:** additive minor bump (→ 0.2.x / 0.3.0)
- **Depends on:** nothing. **Blocks:** ChapterX portal-connector attachment sends (P2 images, P4 files).

---

## 1. Summary

Let a client attach a file by **inline bytes** (base64), not only by a
filesystem **path**. Add `bytes?` (and optional `contentType?`) to `OutgoingFile`,
make `path?` optional, and require exactly one of `{path, bytes}`. The relay
decodes inline bytes straight into a Discord `AttachmentBuilder`.

This unblocks remote clients that hold generated bytes in memory, and it closes a
filesystem-disclosure hole in the current path-only design.

## 2. Motivation

Two problems, one fix.

**(a) Remote clients can't upload.** `OutgoingFile` today is `{ path, name?,
description? }`, and the **relay** reads that path off **its own** disk
(`discord-bot.ts:666` `buildAttachments` → `new AttachmentBuilder(f.path, …)`).
So a file can only be attached if it already exists on the relay host. The
driving client — **ChapterX as a portal client** — runs as a *separate* process
(potentially a different host) and produces attachments **in memory**:
model-generated images (base64) and downloaded files (Buffers). It has no way to
hand those to the relay. ChapterX's P1 connector currently degrades these to
caption-only text and logs a warning; this RFC is what makes real image/file
sends possible.

**(b) Path-based files are a filesystem-disclosure hole.** Because the relay
reads any client-supplied path, a client can request `{ path:
'/etc/passwd' }` (or any readable file on the relay host) and receive it back as
a Discord attachment. For a shared, multi-tenant relay that is an arbitrary
file-read primitive. Inline bytes remove the relay-side read entirely; the client
can only attach data it already has.

## 3. Background — how it works today

- Wire type (`portal-protocol/src/rpc.ts`):
  ```ts
  export interface OutgoingFile { path: string; name?: string; description?: string }
  ```
- `send_message` carries `files?: OutgoingFile[]` (`SendMessageParams`).
- The relay passes them straight through: `relay.ts` → `webhooks.send({ …, files:
  p.files })` → `discord-bot.ts:212` `buildAttachments(opts.files)`:
  ```ts
  if (!f?.path || !existsSync(f.path) || !statSync(f.path).isFile()) throw …
  const a = new AttachmentBuilder(f.path, { name: f.name || basename(f.path) })
  ```
- `MAX_ATTACH = 10` (`discord-bot.ts:114`).
- The gateway WS server sets **no `maxPayload`** (`gateway.ts:103`,
  `new WebSocketServer({ port, host: '127.0.0.1' })`), so the `ws` default cap of
  **100 MiB** per message already applies — base64 attachments fit comfortably
  within typical Discord upload tiers (8/25/50/500 MB).

## 4. Problem statement

`OutgoingFile` is path-only and relay-read. That (a) makes attachments impossible
for any client that isn't co-located with the relay and holding files on the
relay's disk, and (b) exposes the relay host's filesystem to clients.

## 5. Proposed change

### Protocol (`portal-protocol/src/rpc.ts`)

```ts
/** A file to upload. Provide EXACTLY ONE source:
 *  - `bytes`: base64-encoded content (preferred; works from any client/host).
 *  - `path`:  a path the RELAY can read (co-located/trusted clients only). */
export interface OutgoingFile {
  /** Display filename. Required when using `bytes` (no basename to infer). */
  name?: string;
  /** Base64-encoded file content. */
  bytes?: string;
  /** A path on the relay host. Discouraged — see security note. */
  path?: string;
  /** Optional MIME type (Discord mostly infers from `name`). */
  contentType?: string;
  description?: string;
}
```

Validation (relay-side, per file): exactly one of `bytes`/`path` set; when
`bytes` is set, `name` is required. Reject otherwise with `INVALID_PARAMS`.

### Relay (`portal-relay/src/discord-bot.ts` — `buildAttachments`)

```ts
return files.map((f) => {
  if (f.bytes != null) {
    if (!f.name) throw new Error('inline file requires a name')
    const buf = Buffer.from(f.bytes, 'base64')
    if (buf.length > MAX_INLINE_BYTES) throw new Error(`File too large (${buf.length} > ${MAX_INLINE_BYTES})`)
    const a = new AttachmentBuilder(buf, { name: f.name })
    if (f.description) a.setDescription(f.description)
    return a
  }
  if (f.path != null) {
    if (!allowPathFiles) throw new Error('path-based files are disabled on this relay')
    // …existing existsSync/statSync/AttachmentBuilder(path) path…
  }
  throw new Error('file requires `bytes` or `path`')
})
```

### Relay config (`portal-relay/src/config.ts`)

- `maxInlineFileBytes` (default e.g. **8 MiB** — the Discord non-boosted ceiling;
  raise per the relay bot's tier). Decoded-size check, as above.
- `allowPathFiles` (default **false**). Path-based files become opt-in, since
  they are the disclosure vector. Co-located/trusted deployments can re-enable.
- Optionally set an explicit `maxPayload` on the `WebSocketServer`
  (`gateway.ts`) so the WS-frame cap and `maxInlineFileBytes` are coherent (a
  frame must hold base64, ~1.34× the raw bytes, plus the rest of the message).

### Client ergonomics (`portal-client`)

Keep `sendMessage` a passthrough, but add a tiny helper so callers with a Buffer
don't hand-roll base64:

```ts
export function fileFromBytes(name: string, data: Buffer | Uint8Array, opts?: { contentType?: string; description?: string }): OutgoingFile
// → { name, bytes: Buffer.from(data).toString('base64'), ...opts }
```

ChapterX then maps its in-memory image/file Buffers directly:
`files: [fileFromBytes(filename, buffer, { contentType })]`.

## 6. Limits & security

- **Size:** enforce `maxInlineFileBytes` on the *decoded* length; surface a clear
  `INVALID_PARAMS`/oversize error rather than letting Discord reject it opaquely.
  Keep `MAX_ATTACH = 10`.
- **Path files default-off:** flipping `allowPathFiles` to false by default closes
  the arbitrary-read hole for shared relays while preserving the feature for
  trusted single-tenant setups. (If preferred, keep default-on but log loudly —
  the RFC recommends default-off.)
- **No streaming (yet):** base64-in-frame is simple and covers normal Discord
  attachments. Very large or streamed uploads (chunked/binary frames, or a side
  HTTP upload returning a handle) are explicitly out of scope; revisit if a use
  case needs >~tens of MB.

## 7. Compatibility

- **Additive.** Existing `{ path }` callers keep working **iff** `allowPathFiles`
  is enabled. If we ship `allowPathFiles:false` by default, that is the one
  behavioral change — call it out in the changelog; flip it on for any relay that
  relied on path files.
- `name` becomes optional in the type but is **required when `bytes` is used**
  (enforced at runtime, since TS can't express xor cleanly here).

## 8. Testing

- **Unit (`buildAttachments`):** `bytes` → `AttachmentBuilder` from a Buffer with
  the given name; missing-name rejected; oversize (> `maxInlineFileBytes`)
  rejected; `path` with `allowPathFiles:false` rejected; both/neither rejected.
- **Live (mirrors `scripts/p1-live.mjs`):** send a small PNG via `bytes` to
  `#test`, assert the resulting message carries one attachment whose name/size
  match; round-trip an inbound fetch to confirm it surfaces as a `PortalAttachment`.
- **ChapterX integration:** once landed, flip the P1 image/file degradations to
  real `fileFromBytes` sends and assert a generated image posts.
